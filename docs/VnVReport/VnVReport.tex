\subsection{Area of Testing 2: Attendee Application}
This section evaluates the functional requirements specific to the attendee's mobile experience. Testing focuses on the end-to-end journey for students, including registration workflows, event discovery, and the feedback submission process.

\subsection{Area of Testing 3: System Integration and Security}
This section verifies the robustness of the system's internal communications and its defensive posture. Testing includes real-time data synchronization between the admin and attendee interfaces, as well as manual verification of role-based access controls to prevent unauthorized data access.

\subsubsection{Test-FR-SI-1: Real-Time Data Synchronization}
\textbf{Type:} Automatic \\
\textbf{Initial State:} Admin and attendee interfaces are active. \\
\textbf{Input/Condition:} Admin updates event details (e.g., venue or date). \\
\textbf{Expected Output:} Attendee view updates automatically without manual refresh. \\
\textbf{How the test will be performed:} A script will trigger a PATCH request to the event endpoint and then verify if the WebSocket listener on the client-side received the payload and updated the local state. \\
\textbf{Actual Result:} The attendee UI updated within 200ms of the admin change, reflecting the new location details without a page reload.

\subsubsection{Test-FR-SI-2: Role-Based Access Verification}
\textbf{Type:} Manual \\
\textbf{Initial State:} Admin and attendee accounts exist with distinct permissions. \\
\textbf{Test Procedure:} \\
\textbf{Input:}
\begin{itemize}
    \item Login with an "Attendee" role account.
    \item Attempt to access the Admin Dashboard URL manually.
    \item Attempt to trigger a "Create Form" API request using the attendee token.
\end{itemize}
\textbf{Expected Result:} \\
\textbf{Output:}
\begin{itemize}
    \item The system denies access to admin features.
    \item Proper error handling (e.g., 403 Forbidden or redirection) is observed.
\end{itemize}
\textbf{Result:} Pass \\
\textbf{Discussion:} The system successfully enforced role-based access control, ensuring that sensitive administrative functions are strictly limited to authorized accounts.

\subsubsection{Test-FR-SI-3: API Security - SQL Injection}
\textbf{Type:} Automatic \\
\textbf{Initial State:} System API is live and accepting search queries. \\
\textbf{Input/Condition:} Submit a string containing SQL commands (e.g., "OR 1=1") into the event search bar. \\
\textbf{Expected Output:} The system treats the input as a string and returns no results; no data leak occurs. \\
\textbf{How the test will be performed:} An automated security suite will run a payload of common SQL injection strings against all search endpoints to check for database error leaks or unauthorized data retrieval. \\
\textbf{Actual Result:} All payloads were successfully sanitized or rejected; the database remained secure.

\section{Changes Due to Testing}

Reviews from peers and our supervisor revealed many issues with the first revision of the software, including usability issues and bugs that are difficult to catch with regular unit testing or were simply overlooked during manual testing by developers.

\subsection{Server Robustness Improvements}

Based on the results from unit testing, some changes can be made to improve server robustness. For example, TC69 verifies duplicate registration prevention, but enforcing uniqueness constraints at the database level on (userId, eventId) would provide stronger protection. Additionally, TC3 demonstrates graceful error handling, however, implementing centralized logging and monitoring would improve debugging and system reliability during unexpected failures.

\subsection{Rendering Bug Fixes}

During the reviews, a couple minor bugs were found mostly related to the initial rendering of pages and transfer of parameters between pages of the web application.

\subsubsection{User Portal}
\begin{itemize}
    \item
    \textbf{Bug:} Clicking the "Back to Surveys" buttons within the form filling interface brings the user back to the dashboard page, but on the "Overview" tab, not the "Surveys" tab.
    \\
    \textbf{Fix:} Each tab (Overview, Events, Surveys) can be attached to a URL route (e.g. /overview, /events, /surveys).
    \item
    \textbf{Bug:} Clicking "Save" under the form filling interface, then reloading the page does not retain the user's current answers.
    \\
    \textbf{Fix:} Ensure the database is queried for any saved responses to the selected form, and the reponses are loaded before the page is rendered.
    \item
    \textbf{Bug:} The modular form filling interface does not render the "Completed Sections" interface until all modules are completed.
    \\
    \textbf{Fix:} Change the rendering conditions for the "Completed Sections" UI element to render if there are 1 or more completed sections.
    \item
    \textbf{Bug:} Clicking "View Details" on an event card under the "Featured Events" section loads the overall "Events" tab, instead of bringing the user to the details page for the selected event.
    \\
    \textbf{Fix:} Update the click handler for the "View Details" button to route to the event details page.
\end{itemize}

\subsubsection{Admin Portal}
\begin{itemize}
    \item
    \textbf{Bug:} Setting the unlock date on a form to an invalid date will set no date instead of prompting the user to correct the invalid entry.
    \\
    \textbf{Fix:} Add error handling for the selected unlock date, with validity checks that trigger a correction prompt on invalid entries.
    \item
    \textbf{Bug:} Empty sections (e.g. form with no description) are not rendered, causing inconsistencies in UI elements and visual bugs
    \\
    \textbf{Fix:} Add placeholder text for missing information (e.g. No description, no deadline) or render the element without any text.
\end{itemize}

\subsection{Missing Features}

The following are important features or functionality of the software that was noted to be missing by reviewers due to oversights during development.

\subsubsection{User Portal}
\begin{itemize}
    \item
    \textbf{Issue:} No option to view a completed submission, only to edit or delete.
    \\
    \textbf{Fix:} Render a "View Submission" button if the form is completed. When clicked it should open the form filling interface but disable all input fields to prevent edits.
    \item
    \textbf{Issue:} The form details interface displays the time the form was created, which is irrelevant to the user. Should display the last edit or submission instead.
    \\
    \textbf{Fix:} Populate this field with the last submission time, queried from the form submissions database table.
    \item
    \textbf{Issue:} Instead of prompting the user for their personal information (e.g. name, email), this information should be pulled from the user's profile.
    \\
    \textbf{Fix:} Create a user profile page, allowing the user to edit this information. This page can be rendered and required for completion upon creation of a new account.
\end{itemize}

\subsubsection{Admin Portal}
\begin{itemize}
    \item
    \textbf{Issue:} No option to edit the name and description of a form
    \\
    \textbf{Fix:} Similar to the events dashboard, add an edit button for each form which prompts the admin to edit the name and description of the form.
    \item
    \textbf{Issue:} The status of an event ("Scheduled", "Ongoing", or "Completed") should be an automatically assigned state instead of manually selected.
    \\
    \textbf{Fix:} Add functionality in the backend to do this automatically at the indicated timestamps (e.g. set to "Ongoing" when the event date and time is reached).
    \item
    \textbf{Issue:} Cannot set the status of an attendee from "Registed" to "Attended"
    \\
    \textbf{Fix:} This can be done by scanning the user's QR code, but should add an interface to set this manually
    \item
    \textbf{Issue:} The dropdown question type in the form builder should have the option to populate the dropdown with options by pasting multiple options seperated by newlines, instead of having to manually enter each choice for larger dropdowns.
    \\
    \textbf{Fix:} Add a seperate field to enter multiple options at once, or detect new line characters in the existing interface.
    \item
    \textbf{Issue:} Pressing the Enter key while adding options to a multiple selection question should create a new option, instead of having to use the UI buttons.
    \\
    \textbf{Fix:} Add detection for new lines in the option field, or an event handler for the enter key.
\end{itemize}

increments, and it made more sense to validate features as they were
built. In future projects, we would anticipate this by planning for more
continuous testing throughout development rather than assuming testing
would happen mostly at the end.
