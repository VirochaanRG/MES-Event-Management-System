\section{Extras}

We chose the Usability Report and Norman's Design Principles as our two Extras. These were chosen because the product is a human-facing event management platform used by two distinct user groups, and the overall project risk was more strongly tied to usability, task flow clarity, and overall good design and user experience.

\subsection*{Usability Report}

The Usability Report extra involved conducting a structured evaluation of the system with representative users to assess how effectively both student attendees and administrative organizers could complete their primary workflows. This extra was chosen because the MES platform is inherently interaction-driven: its value depends entirely on users being able to register for events, complete forms, and manage event logistics without any confusion. A functional system that users struggle to navigate still fails to meet its core purpose. \newline
\newline
The evaluation followed a task-based methodology. Participants were recruited from both user groups and asked to complete a set of predefined end-to-end workflows covering the most critical parts of the system, in both the user and admin interfaces. Participants were observed while working through each task using a think-aloud protocol, which allowed the evaluators to capture points of hesitation, confusion, and incorrect navigation paths as they occurred rather than relying solely on post-hoc recall.\newline
\newline
Following the task sessions, participants completed a structured survey to provide quantitative feedback on perceived ease of use, interface clarity, and overall satisfaction. Task completion rates and the nature of any errors were recorded and analyzed alongside the qualitative observations. The findings from these sessions directly informed several interface refinements made during the final development phase, particularly around form navigation clarity and the visibility of key actions within the admin dashboard.

\subsection*{Norman's Principles}

The Norman's Principles extra applied Don Norman's foundational interaction design framework to critically evaluate the user interface decisions made throughout the project. This extra was chosen because it offered a principled basis for design critique that goes beyond subjective visual preference or informal testing impressions.The evaluation worked through each principle in the context of the platform's two interface surfaces: the student-facing web application and the admin dashboard. 

The value of this extra was that it grounded design decisions in an established theoretical framework rather than purely in testing outcomes. Where usability testing revealed that something was difficult to use, the Norman's Principles analysis provided us with a different outlook and allowed us to approach the design from a different perspective.



\section{Design Iteration}

Our front-end design underwent revisions based on supervisor feedback, who
evaluated the product as an end-user for both attendee and admin portals as
part of our usability testing. The UI for specified sections was revamped to
address the concerns raised.

The underlying architecture remained the same as the client was clear and
specific about how the system should interact with the environment and how
data should be stored. As a result, it was possible to satisfy requirements
with the initial architecture.

\subsection{Usability Testing}

Usability testing directly influenced our UI design,

\subsubsection{User Profile Configuration}

\begin{itemize}
    \item \textbf{Problem:} Manual re-entry of user details for every event is
    redundant for participants and creates administrative overhead when
    designing registration forms.
    \item \textbf{Solution:} Implement a mandatory user profile system that
    automatically populates relevant fields in registration tables, streamlining
    the enrollment process.
\end{itemize}

\subsubsection{Form Builder Survey Creation Automation}

\begin{itemize}
    \item \textbf{Problem:} Manual form construction is labor-intensive and
    error-prone, particularly when managing large-scale surveys.
    \item \textbf{Solution:} Support CSV template uploads to automate form
    generation, significantly reducing development time and ensuring data
    consistency.
\end{itemize}

\subsubsection{UI Consistency}
\begin{itemize}
    \item \textbf{Problem:} Fragmented UI Components and Aesthetic Inconsistency
    \begin{itemize}
        \item Disparity between custom UI library components and native
        browser-default form elements (e.g., buttons, inputs, and dropdowns).
        \item Non-uniform colour application and lack of a cohesive visual
        identity across different modules.
    \end{itemize}
    
    \item \textbf{Rationale:} Enhanced Platform Literacy and User Intuition
    \begin{itemize}
        \item A standardized interface reduces cognitive load, allowing users to
        predict behavior based on established visual patterns.
        \item Consistent design strengthens brand identity and provides a more
        professional, polished user experience.
    \end{itemize}
    
    \item \textbf{Solution:} Global Component Standardization
    \begin{itemize}
        \item Refactor all modals and interactive elements to strictly utilize
        the primary UI library.
        \item Enforce a unified design system for state-dependent styling,
        specifically for:
        \begin{itemize}
            \item High-contrast borders for selected or focused elements.
            \item Standardized hover states and active press feedback.
            \item Consistent padding, shadows, and corner radii across all containers.
        \end{itemize}
    \end{itemize}
\end{itemize}

\begin{itemize}
    \item \textbf{Problem:} Lack of Visual Feedback for Multi-Step Workflows
    \begin{itemize}
        \item Users have no clear indication of completion status when
        navigating modular surveys or multi-page forms.
        \item Absence of "save-and-continue" context leads to uncertainty and
        higher abandonment rates.
    \end{itemize}
    
    \item \textbf{Rationale:} Cognitive Mapping and Completion Motivation
    \begin{itemize}
        \item Visualizing progress provides a "mental map" of the task, reducing
        the perceived effort of long forms.
        \item Constant feedback reinforces a sense of accomplishment,
        encouraging users to finish the registration or survey.
    \end{itemize}
    
    \item \textbf{Solution:} Integrated Progress Tracking System
    \begin{itemize}
        \item Implement a dynamic progress bar or "stepper" component at the top
        of modular interfaces.
        \item Enhance the UI with specific status indicators, including:
        \begin{itemize}
            \item \textbf{Percentage Completion:} A real-time calculation of
            answered versus total required fields.
            \item \textbf{Sectional Checkmarks:} Visual confirmation when an
            entire module or category is successfully validated.
            \item \textbf{Time-to-Complete Estimates:} Providing an approximate
            remaining duration based on average user interaction.
        \end{itemize}
    \end{itemize}
\end{itemize}

\section{Design Decisions}

 \subsection{Limitations}
The system is built on a synchronous REST API architecture, which means it does not support real-time data propagation. Changes made by an admin are not automatically propogate meaning users must refresh or re-navigate to see updated state, which is a functional boundary of the current implementation rather than a deliberate design preference.

The form builder supports conditional section unlocking based on prior responses, but this branching logic is linear. It cannot support complex non-linear branching where multiple prior responses across different sections combine to determine which content is shown next. This limits the expressiveness of surveys that require more sophisticated conditional flows.

 \subsection{Assumptions}
Certain assumptions about event-day proceedings have influenced how we track and
show attendance statuses for event participants. In the current system, it is
assumed that once attendees have checked-in at the event, they are given a
physical indicator of their attendance on-site, such as a wristband or a stamp.
As a result, there is no software indicator in the attendee portal which can be
used to demonstrate to event administrators that the user has checked-in to the
event. This is a decision we are happy to make as it reduces the workload
required for an MVP, and there is a system in place already which does not need
to be replaced which addresses this issue.

\subsection{Constraints}
\subsubsection{Technological Constraints}
The technological constraints determined by our client influenced our design,
especially our backend and data architecture. The tech stack constraints
required us to use JavaScript and TypeScript-based backend frameworks and
methodologies to implement our APIs and routing. Our data needed to be formatted
and stored in a relational database as we used a SQL DBMS. These constraints
also required us to deploy using our client's hosted server as it was compatible
with the deployment technology used for our backend. As a result of using JS and
TS-based frameworks, error handling and complete case coverage was less prioritized
where possible in order to deliver an MVP faster.

The requirement to implement a bespoke form builder and form submission system
meant we could not use any pre-existing open-source form libraries. This
required us to design how modular forms interact with each other, how they
stored with respect to the relational database, and how they are rendered to the
user. We decided to implement modular surveys as a series of independent
surveys, with sections unlocking if necessary based on specific responses or the
completion of previous sections. This ensured our form builder was meaningfully
different from existing form-building software, and provides the client with the
flexibility to reuse form results and make surveys digestible for attendees when
they complete them.

\subsubsection{Scheduling Constraints}
The initial and revised scheduling constraints influenced our MVP features. We
decided to forego a fully-integrated login system with the McMaster ecosystem as
specific features such as the ability to purchase multiple tickets or transfer
tickets to other users. We recognized that these changes are trivial compared to
the features we have deployed in our MVP, so pushing it in the backlog made sense
from a continuous and iterative development process perspective.

\subsubsection{Expandability \& Integration Constraints}
One of the stretch goals of this project was to provide this product as a SAAS
product for organizations outside of the McMaster Engineering Society to use.
The initial plan of the software was also to be integrated with software from
other groups within the same ecosystem, though this later changed. As a result,
our API and database design remained mostly detatched from McMaster-specific
infrastructure, making it more portable. This design decision helped with
abstraction of the system, and made front-end integration simpler as API routes
were well-defined.

\section{Economic Considerations (LO23)}

There is a clear market for EvENGage, particularly among universities, student organizations, and event-based clubs that require a centralized event management system. The product can be offered using a Software-as-a-Service (SaaS) model, where organizations subscribe to access the platform. Pricing could be tiered based on the number of events, users, or advanced features such as analytics and custom integrations. Initial costs would include server hosting, maintenance, and future development/troubleshooting. There are several SaaS formats that we could follow, so the setup process would be relatively straightforward. Moreover, we designed the system with scalability in mind, so the cost and time required to extend to additional clients is small.

The process of attracting users would involve starting small, and reaching out to student clubs and organizations. Given that the product is already designed for MES, it has the potential to be adapted for use by other organizations within McMaster. From there, there is a large market of other universities with similar needs.

We ensured that after we got into development though we had crossover in some of the responsibilities so team members could get experience in other areas of development as well to improve our overall experience during this project. The project management roles assigned were also effective in ensuring we stayed on track during meetings and when completing deliverables.
\newline\newline
The workflow plan we used was essentielly followed except we decided to forgo making unit tests on each pull request and decided this would be done during the testing stage of the project.
\newline\newline
Our expected technology essentielly stayed the same as it was constrained upon us by our supervisor. The one change that we decided was that instead of Next.Js we would use Vite which is a more lightweight framework allowing for faster development and deploying. To accompany that we had to use Vitest as our testing library in conjunction with Jest.

\subsection{What Went Well?}

The team's communication structure worked effectively throughout the project. Discord provided a low-friction channel for day-to-day discussion, while GitHub Issues ensured that action items from meetings were captured and traceable. The Kanban board gave the team a shared, up-to-date view of task status, which reduced ambiguity about who was responsible for what and what remained to be done. Switching from Teams to Discord I think was a good decision as it was a tool we had more familiarity with and also gave us quicker communication times and faster and easier ways to meet and call quickly. \newline\newline
When looking at it from a process point of view having standardized requirements very early on made it super easy for us to develop and since they didn't change it gave us ample time to implement the. Having solid requirements was a real boon in this project and having good communication with our supervisor who could help clarify or flesh out requirements helped us immensely during the development process.
\newline\newline
The division of roles based on individual interest and prior experience also proved beneficial. Because team members self-selected into areas they were comfortable with, onboarding to the relevant technologies was faster and the quality of output in each area was higher. Even though there was some crossover in who worked on what having the roles at the start made it easier to start the development process. The designated Reviewer role ensured that deliverables were consistently checked before submission, which improved overall quality. CI/CD through GitHub Actions was effective once configured, automatically enforcing code style and running tests on code being merged in. Though this was a new tool for many of us we found that it was a good decision to utilize it as it helped stop many errors in the long run.
\newline\newline
The tech stack that we used was also very useful. It got us up and running quite quickly and was very easy to develop and debug with. Specifically the choice of Vite though imposed on us was a blessing in disguise as we managed to develop quickly with the help of the Hot Reload functionality allowing us to see the changes in real-time. Using built in styling libraries through TailWind also saved us a lot of time and made our design look better and more consistent.
\subsection{What Went Wrong?}

The initial setup of the CI/CD pipeline required more time and effort than anticipated. Team members with less prior exposure to GitHub Actions found the configuration process unintuitive, and debugging failed workflow runs introduced delays early in the development phase. This overhead temporarily slowed progress on technical deliverables.
\newline\newline
The complexity of the form builder and event registration components was consistently underestimated at the planning stage. Both features required significant custom design work: the form builder needed to support modular, reusable survey sections with conditional unlocking logic, and the event registration system needed to integrate QR code generation with attendance tracking. We wanted to have this ready for Rev0 but since we built this in-house it wasn't fully ready. This extended the development timeline for both components well beyond initial estimates, and required revisiting some architecture decisions partway through implementation.
\newline\newline
Automated testing was not integrated into the development workflow as early as it should have been. Because tests were written primarily toward the end of the project rather than alongside feature development, some edge cases and regressions were caught later than they could have been. This also meant that the CI/CD pipeline was not fully leveraged for quality assurance during the most active development period, reducing its practical value until the project was nearly complete.

\subsection{What Would you Do Differently Next Time?}

For future projects, the team would invest more time upfront in configuring and testing the CI/CD pipeline before development begins in earnest, rather than setting it up incrementally alongside feature work. Also having testing better integrated with the workflow and development process would probably more akin to a real world environment and provide us with an easier time integrating new changes. From a technical development perspective, we would have liked to have all our features done in a crude implementation for our POC or Rev0 demonstrations so we could expand or add more features for our Rev1. This would have really helped us push and try and achieve the strech goals we set at the start of the project.
\newline\newline
Also in general we would like to have better team coordination and communication. Though it wasn't a big obstacle to our success having more consistent formal meetings rather than our informal discussions could have proven useful in ensuring that we all on the same track. Doing this would have also helped ensure we could track progress more effectively and been a bit more organized.



\section{Ethics and Equity (LO18)}

Our platform was designed with accessibility and equity as core principles.
